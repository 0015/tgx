69 /***********************************************************************************************
70 *
71 * Image<color_t> : Template class for an "image object" that draws into an external memory buffer.
72 *
73 * The class object itself is very small (16 bytes) thus image objects can easily be
74 * passed around/copied without much cost. Also, sub-images sharing the same memory
75 * buffer can be created and are useful for clipping drawing operation to a given
76 * region of the image.
77 *
78 * *** NO MEMORY ALLOCATION IS EVER PERFORMED BY THE CLASS OR ITS METHOD. ***
79 *
80 * -> Nothing happens under the hood;It is the user's responsability to assign the buffer to the image.
81 *
82 * Template parameter:
83 * -------------------
84 *
85 * color_t: Color type to use with the image, the memory buffer supplied should have type color_t*
86 * and correct alignement. It must be one of the color types defined in color.h
87 * Typical choice: -> color_t = RGB32 when running on a desktop computer.
88 * -> color_t = RGB565 when running on an MCU with limited memory (Teensy, ESP32..)
89 *
90 * Image layout:
91 * -------------
92 * The image in the memory buffer in made of consecutive pixel of type color_t in row major
93 * order but with a stride which may be larger than the image width (i.e. row lenght).
94 *
95 * More precisely, Pixel are ordered from the top left (0,0) to the bottom right (_lx-1, _ly-1)
96 * and the position of pixel (x,y) in the buffer is given by the formula:
97 *
98 * color(x,y) = buffer[x + y * _stride].
99 *
100 * The only thing that the Image object remembers is (_lx, _ly, stride) and the buffer pointer.
101 *
102 * Remarks:
103 * --------
104 *
105 * (1) When creating a new image, one can (and usually should) choose _stride = lx. However, allowing
106 * other stride in the class definition allows to very efficiently represent sub-images without
107 * making any copy and thus enables fast clipping of all drawing operations (sub-images are similar
108 * to Numpy view for ndarray).
109 *
110 * (2) Avoid using HSV as color type for an image: it has not been tested to work and will be very slow.
111 *
112 * (2) The library defines many drawing primitive and the user can choose between fast or high quality
113 * rendering. The naming of the method follow the convention:
114 *
115 * [draw/fill][Thick/-]MethodName[AA/-]
116 * where:
117 * [draw] means that the method draw a path whereas [fill] means that the method create a solid
118 * shape.
119 * [Thick] means that the outline of the shape/path has user defined thickness and is not just a
120 * single pixel wide. The tichness is usually a float so it does not need to be an integer
121 * but it should be at least 1 to get good rendering.
122 * [AA] means that the method is 'high quality': it is drawn with anti-aliasing and sub-pixel
123 * precision. Method with this suffix take fVec2 parameters for coordinates inputs whereas
124 * regular 'low quality' methods (without the AA suffix) take iVec2 parameters.
125 *
126 * (3) Opacity and alpha blending.
127 * All drawing methods that take a color as parameter also take an 'opacity' float parameter which
128 * allows to create transparency effect even when color type used do not have an alpha channel
129 * (for instance, RGB32 as an alpha channel but RGB565 does not).
130 * - Passing TGX_DEFAULT_NO_BLENDING or (a negative value) for the opacity will usually disable
131 * blending the method will simply use overwritting of the destination pixel (which may be a bit
132 * faster but not much).
133 * - Passing 0.0f < opacity <= 1.0f will enable blending and the alpha channel of the colors will
134 * be used if present.
135 * Remark: using opacity=TGX_DEFAULT_NO_BLENDING and opacity=1.0f gives the same result if the
136 * color in play are opaque (which is the case when the color type has no alpha channel)
137 * but give different result if the color are semi-transparent.
138 *
139 * (4) As indicated in color.h. All colors with an alpha channel are assumed to have pre-multiplied
140 * alpha and treated as such for all blending operations.
141 * cf: https://en.wikipedia.org/wiki/Alpha_compositing
142 *
143 * (5) Most methods require iVec2/fVec2 parameters to specify coordinates in the image. Using initializer
144 * list, it is very easy to call such method without explicitly mentioning the iVec2 type. For example,
145 * a function with signature f(iVec2 pos, int r) can be simply called f({x,y},z)' which is equivalent
146 * to the expression f(iVec2(x,y),z)
147 *
148 * Note also that iVec2 vector are automaticcally promoted to fVec2 vector so that calling AA method
149 * with sub-pixel precision is effortless even when only working with integer vectors iVec2. On the
150 * other hand, downgrading from fVec2 to iVec2 necessitate an explicit cast/conversion.
151 *
152 * (6) Creating sub-image is costless and enables to restrict any drawing operation to any rectangular
153 * region of an image. This is a very powerful method: for example, it permit to draw half and quarter
154 * circles from the full drawCircle method !
155 *
156 **********************************************************************************************/
